# ADLockoutBuster — How It Works

## The Lockout Problem

Active Directory uses a concept called **bad password count**. Every time a user enters a wrong password, the DC increments this counter. When the counter hits the domain's **Account Lockout Threshold** (e.g. 5 bad attempts), the account is locked.

The tricky part: **the bad password count resets on password change, but saved credentials don't**. If a user changes their password, any device, service, or application that still has the old password cached will keep hammering the DC until the account locks — usually within seconds.

---

## The Two Critical DCs

### PDC Emulator — Where Lockouts Are Recorded

In a multi-DC domain, **Event 4740 (Account Locked Out)** is always written on the **PDC Emulator** (regardless of which DC actually processed the last bad password). This event contains the `CallerComputerName` field — the machine that sent the bad password.

This is why ADLockoutBuster prioritises the PDC Emulator and why Microsoft's guidance says "check the PDC first."

### All DCs — Where Failed Logons Are Recorded

**Event 4625 (Failed Logon)** is written on *whichever DC processed the bad authentication attempt*. In large environments with multiple DCs, you need to scan all of them — a different machine may be hitting a different DC each time.

---

## Event Flow for a Lockout

```
User changes password on DC1
    ↓
Phone with Exchange ActiveSync still has old password
    ↓
Phone → Exchange Server → NTLM auth to DC2 → Bad password (Event 4625 on DC2)
Phone → Exchange Server → NTLM auth to DC3 → Bad password (Event 4625 on DC3)
Phone → Exchange Server → NTLM auth to DC1 → Bad password → threshold hit
    ↓
DC1 locks the account → Replicates lock to PDC Emulator
    ↓
PDC Emulator writes Event 4740 with CallerComputerName = Exchange Server
```

The exchange server appears as the "caller" because it is proxying the auth — but the real source is the phone. ADLockoutBuster shows the Exchange server IP so you can correlate it with Exchange ActiveSync logs.

## How ADLockoutBuster Queries Events

### PowerShell-Based Event Collection

All event log access uses PowerShell's `Get-WinEvent` with XML filter queries. This approach was chosen because:

1. **No additional dependencies** — PowerShell is built into all supported Windows versions
2. **Remote collection works the same as local** — just add `-ComputerName`
3. **XML filters are fast** — the filter runs on the remote machine before data is sent
4. **Handles auth gracefully** — uses the running user's Kerberos ticket

```powershell
# Example query generated by the engine
Get-WinEvent -ComputerName 'dc01.corp.local' -FilterXml @'
<QueryList>
  <Query Id="0" Path="Security">
    <Select Path="Security">
      *[System[(EventID=4625 or EventID=4740 or EventID=4771)
       and TimeCreated[@SystemTime>='2025-01-01T00:00:00.000Z']]]
    </Select>
  </Query>
</QueryList>
'@
```

Each event is returned as XML, then parsed into structured `LockoutEvent` objects.

### Event Field Extraction

For each event, the engine extracts:

| Field | Event XML Path | Meaning |
|---|---|---|
| `TargetUserName` | `EventData/Data[@Name='TargetUserName']` | The account being authenticated |
| `WorkstationName` | `EventData/Data[@Name='WorkstationName']` | NetBIOS name of source machine |
| `IpAddress` | `EventData/Data[@Name='IpAddress']` | Source IP address |
| `LogonType` | `EventData/Data[@Name='LogonType']` | How the logon was attempted |
| `AuthenticationPackageName` | `EventData/Data[@Name='AuthenticationPackageName']` | NTLM vs Kerberos |
| `Status` | `EventData/Data[@Name='Status']` | NTSTATUS failure code |
| `SubStatus` | `EventData/Data[@Name='SubStatus']` | More specific failure code |
| `ProcessName` | `EventData/Data[@Name='ProcessName']` | Process that attempted auth |
| `CallerComputerName` | `EventData/Data[@Name='CallerComputerName']` | Machine that initiated (4740) |

## NTSTATUS Code Decoding

When a logon fails, Windows records an NTSTATUS code. ADLockoutBuster decodes these into human-readable reasons:

| Code | Meaning | Implication |
|---|---|---|
| `0xC000006A` | Wrong Password | Account exists, password is wrong |
| `0xC0000064` | Unknown Username | Possible spray attack or mistyped username |
| `0xC0000234` | Account Locked Out | The account was already locked when this attempt came in |
| `0xC0000071` | Password Expired | Password expired — user needs to change it |
| `0xC000006F` | Invalid Logon Hours | Login outside allowed hours |
| `0xC0000072` | Account Disabled | Account is disabled |

When the **SubStatus** is `0xC0000234` (Locked Out), the attempt itself didn't cause the lockout — the account was already locked. This means you should look at events *before* this timestamp.

## Spray Attack Detection Algorithm

The `detect_spray_attacks` method uses a **sliding window** algorithm:

```python
for each source IP:
    sort all events by timestamp
    slide a window of N minutes across the events
    if distinct_usernames_in_window >= threshold:
        flag as spray
```

A spray attack is characterized by:
- One source IP
- Many different usernames
- In a short time window

A normal lockout is characterized by:
- Multiple source machines
- One username
- Repeated attempts

## Netlogon Log Parsing

Netlogon debug logging records authentication attempts at the NETLOGON service level, *before* the Security event log. This gives additional detail in cases where:

- The security event log has been cleared
- The caller machine field is blank in 4625 events
- You need the exact time correlation

Netlogon entries look like:
```
2025-01-15 14:32:11 [BADPASSWORD] SamLogon: Transitive Network logon of CORP\john.doe from LAPTOP01 (via DC02) Entered
2025-01-15 14:32:11 [BADPASSWORD] SamLogon: Transitive Network logon of CORP\john.doe from LAPTOP01 (via DC02) Returns 0xC000006A
```

The engine uses regex to extract: timestamp, source machine, username, and the result code.

## Service & Task Detection

When an account locks out repeatedly from the same machine with **Logon Type 5 (Service)** or **4 (Batch)**, the source is almost always a Windows Service or Scheduled Task.

The engine queries:
```powershell
# Services
Get-WmiObject Win32_Service | Where-Object { $_.StartName -like "*username*" }

# Scheduled Tasks
Get-ScheduledTask | Where-Object { $_.Principal.UserId -like "*username*" }
```

This only works on the *local machine*. The intention is that you run ADLockoutBuster on the machine identified as the source, or check the source machine manually.

---

## Reverse DNS for IP Attribution

When a source IP is found, ADLockoutBuster uses `socket.gethostbyaddr(ip)` to resolve it to a hostname. This:
- Identifies "192.168.1.50" as "LAPTOP-FINANCE-01.corp.local"
- Works for internal IPs (split DNS / internal zones)
- Fails gracefully for external IPs or unregistered machines

## Threading Model

ADLockoutBuster uses Qt worker threads to keep the UI fully responsive during long scans:

```
MainWindow (UI thread)
    ├── ScanWorker (QThread)          — PowerShell event collection
    ├── DCDiscoveryWorker (QThread)   — AD DC enumeration
    ├── MonitorWorker (QThread)       — 30-second polling loop
    └── LockedAccountsWorker (QThread) — AD locked account query
```

All `pyqtSignal` connections are thread-safe. The UI only updates when signals are received on the main thread.

## Lockout Source Analysis Logic

The `analyze_lockout_source` method builds a ranked source list:

1. **Group events by source** (machine or IP)
2. **Sort by count** (highest frequency = most likely culprit)
3. **Decode logon types** with plain-English explanations
4. **Detect NTLM vs Kerberos dominance** (NTLM = shares/drives; Kerberos = direct auth)
5. **Generate "Most Likely Causes"** section based on patterns

This is the key section of the Source Summary tab and is the tool's primary value-add over raw event log viewers.
